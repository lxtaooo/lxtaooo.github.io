Java Web

什么是Web开发？

Web通信，指的是两台计算机通过网络进行资源文件共享。

资源文件

1、静态资源文件：文件内容固定，不随请求变化

  • 图片、html、
  • 服务端将资源发送给客户端，有客户端打开该文件。

​ 2、动态资源文件：随着请求不同发生变化

​ java文件

​ 服务端计算机将会调用执行，将运行结构交给客户端。

通信方式

  • C/S结构

    C：client 客户端计算机安装指定客户端程序，可以帮助客户端计算机向服务端发送请求。

    S: server在服务端计算机安装一种特定的资源调度软件，接受特定客户端程序发送请求，将请求需要的资源文件推送给客户端计算机。

    优点：可以保护服务端计算机。可以分摊服务端计算机工作压力

    缺点：增加 客户获得服务的成本；增加服务端维护成本

  • B/S结构

    B：浏览器 可以帮助客户端向任意服务端发送请求

    S：server 在服务端计算机安装一种特定的资源调度软件，接受所有浏览器发送的请求，将请求的资源文件推送给客户端计算机浏览器上

    优点：成本低，便宜

    缺点：无法有效的保护服务端计算机；服务端计算机压力大

网络协议包

  • FTP 文件传输协议
  • HTTP 超文本传输协议
  • B/S结构中只使用HTTP协议

HTTP请求协议包：

  • 请求行：主要包括
    • URL：请求的资源文件地址
    • method：浏览器采用的请求方式
  • 请求头：
    • 浏览器希望获得资源文件类型 html、png
    • 浏览器希望获得资源文件内容依赖字符集
    • 如果浏览器采用GET方式，则本次关联请求参数需要保存在请求头
  • 空白行：隔离层
  • 请求体：
    • 如果浏览器采用POST请求方式，本次关联请求参数需要保存在请求体

HTTP响应协议包:响应协议包可以有资源 调度软件写入，也可以有java命令（response）写入。

  • 状态行：
    • 状态码
  • 响应头：
    • 返回文件类型（doc，text）
    • 返回文件内容依赖字符集，浏览器需要使用这个字符集对接收内容进行解码
  • 空白行：隔离层
  • 响应体：静态资源文件，动态资源文件结果，【以二进制形式】

客户端计算机上浏览器管理

如何命令浏览器向指定网站发送请求：

  • 通过FORM表单命令 <form action="www.baidu.com">
  • 通过超链接命令 <a herf="http://www.163.com">163.com</a>
  • 浏览器地址栏对象 windows.location="http://www.baidu.com";

请求方式【POST/GET】

1、GET方式：

  • 要求浏览器发送请求时，携带请求参数数量不能超过4k
  • 要求浏览器将请求参数写入【请求头】
  • 要求浏览器将请求参数信息在地址栏上显示出来
  • 要求浏览器对接收的资源文件必须缓存到内存

2、POST方式：

  • 不会限制浏览器发送请求时，携带请求参数数量
  • 要求浏览器将请求参数写入【请求体】
  • 要求请求参数信息在地址栏中隐藏
  • 要求浏览器对接收的文件不能保存到缓存【阅后即焚】

如何控制浏览器采用的请求方式

1、让浏览器发送请求时，采用【GET】方式

  • 超链接只能使用get方式
  • location 对象，也只能使用get方式
  • <form action="www.baidu.com " method="get">不写时默认为get

2、采用post方式

<form action="www.baidu.com " method="post">

如何命令浏览器发送请求时，携带指定参数

1、超链接标签

< a href="http://www.baidu.com?uname=mike&age=23">百度</a>

2、location对象

window.location="http://www.baidu.com?uname=sd&sex=man"

以上两种有弊端，请求参数内容是固定的，无法根据客户的具体要求改变

3、表单标签可以接受用户输入的内容，作为请求参数

  • 表单标签分类：input select textarea
  • 表单标签作为请求参数条件
    • 表单标签必须声明在当前FORM标签内部。
    • 表单标签必须声明name，这个属性作为请求参数的名称。
    • 针对checkbox 和 radio ，只有被选中的情况下才可以作为请求参数

<input type="text" name="name"><br/>

<input type="radio" name="sex" value="man">男

<input type="radio" name="sex" value="woman">nv

HTTP服务器

定义

  • 安装在服务端计算机上一种软件
  • 可以解析来自于浏览器发送的 请求协议包
  • 调用服务端上的被请求的资源共享文件
  • 负责生成一个响应协议包
  • 负责将文件内容或文件运行结果写入响应体
  • 负责将响应协议包推送给当前浏览器

HTTP服务器分类：

  • 德国SAP：
  • IBM：weblogic
  • JBOSS：
  • Apache：Tomcat

Tomact：

  • 由JAVA应用程序编写的资源文件调度器
  • 可以调用服务端静态资源文件和动态资源文件
  • Tomcat负责生成响应协议包，负责将响应协议包推送到浏览器上
  • 端口号存储位置：在安装目录下，/conf/server.xml 里面有端口号可以修改

如何创建一个静态资源网站?

1、Tomcat安装目录下/webapps/ 创建一个文件夹，这个文件夹就是网站。

2、在网站中存入静态资源文件

3、启动tomcat

动态网站结构

1、 src：存放JAVA文件

2、web：存放静态资源文件和配置文件

​ WEB-INF/web.xml

servlet 规范

JAVASE和JAVAEE区别

1、javase：java开发基本规范

2、javaee：java企业级开发规范，制定了JVM与13类系统之间交流的规则，javaee有13种，JDBC就是其 中一种，用于设置JVM与所有关系型数据库进行交流的规则。

servlet

1、servlet规范就是JAVAEE规范之一

2、servlet规范中设置了JVM与http服务器之间的交流规则

3、交流规则：

  • 定义了何种Java类可以被Http服务器 来调用
  • 何种Java类可以作为动态资源文件
    • servlet规范中认为只有servlet接口实现类可以作为动态共享资源文件，
    • servlet接口实现类必须有开发人员负责创建
  • Http服务器如何管理动态资源文件

如何开发一个servlet接口实现类

1、 创建一个实现类，需要继承与在 servlet-api中的 javax.servlet.http.HttpServlet类

2、根据来访的浏览器采用的请求方式，重写对应的doGet或者doPost方法

3、在web.xml配置文件中，注册servlet实现类，通知tomcat当前网站中存在一个动态资源文件

  • 新建空工程

  • 配置新tomcat server

  • 新建web模组

  • src中新建servlet

  • package com.lixiaotao.controller;

import java.io.IOException;

/*          如果浏览器采用post方式，向Tomcat索要servlet接口实现类时，
 *          Tomcat调用接口实现类中的doPost方式，并将这个方法运行结果写入响应包中
 *          如果浏览器采用get方式，向Tomcat索要servlet接口实现类时，
 *          Tomcat调用接口实现类中的doGet方式，并将这个方法运行结果写入响应包中
 */

public class Servlet01 extends javax.servlet.http.HttpServlet {
    protected void doPost(javax.servlet.http.HttpServletRequest request, javax.servlet.http.HttpServletResponse response) throws javax.servlet.ServletException, IOException {

    }

    protected void doGet(javax.servlet.http.HttpServletRequest request, javax.servlet.http.HttpServletResponse response) throws javax.servlet.ServletException, IOException {

        System.out.println("servlet接口实现类中doGet方法被调用了");
    }
}

Tomcat调用servlet接口实现类流程

1、浏览器向服务端发送请求

http://localhost:8080/myWeb/first

2、Tomcat接收到请求，首先将请求资源文件作为动态资源文件，到web.xml进行文件定位。

  • 在web.xml中找<url-pattern>/first</url-pattern>标签，在定位<servlet-name>Servlet</servlet-name>

  • 通过name标签，找到<servlet-class>com.lixiaotao.controller.Servlet01</servlet-class>

    找到相应接口实现类

  • 在调用doGet或者doPost方法

3、如果没有则认为是静态资源文件，从web下进行定位。

4、如果还没有则返回404状态码

servlet请求路径写法规则

1、 具体请求路径：

​ 浏览器请求路径与<url-pattern>/first</url-pattern>中的内容完全一致才可以被调用。

2、前置条件请求路径：

​ 如果浏览器请求的资源文件地址以当前servlet中<url-pattern>的内容为开头，利用通配符*

​ <url-pattern>/first/*</url-pattern> 都可以被调用 “/“开头 “*结尾”

3、后置条件请求路径

​ <url-pattern>*.do</url-pattern> 必须用“*”开头

4、请求路径优先级

​ 具体请求路径>前置条件>通配符>后置条件

Servlet实例对象【生命周期】

1、项目中，Servlet的实例对象都是由Tomcat负责创建的，开发人员不能创建实例对象

2、【创建时机】：

  • 【默认情况】：当第一个用户来访问这个Servlet时，Tomcat负责创建它的实例对象
  • 【人工干预】：要求Tomcat在启动时，创建Servlet的实例对象
    • <load-on-startup>1(任意大于0的数)</load-on-startup>

3、在Tomcat运行期间，一个Servlet实现类只能被创建一个实例对象

4、在Tomcat关闭的时候，由Tomat负责销毁所有的Servlet实例对象

Servlet 开发时需要使用的五种工具对象

  • HTTPServletResponse接口：负责将运行结果写入到响应包
  • HTTPServletRequest接口：负责读取请求协议包中的信息
  • ServletContext接口:可以为当前网站中所有的的Servlet提供共享数据
  • cookie：在一次会话过程中，存储浏览器在服务端产生的私人数据
  • Httpsession：存储浏览器在服务端产生的私人数据

HTTPServletResponse接口

  • 一个来自于Servlet规范中的接口；由Tomcat负责提供这个接口的实现类
  • 负责在Servlet运行完毕之后，将相关的数据写入到响应头和响应体。
  • 开发人员习惯将HttpServletResponse接口修饰对象称为响应对象
java

HttpServletRequest接口

  • 一个来自于Servlet规范中的接口；由Tomcat负责提供这个接口的实现类
  • 由HTTPServletRequest接口修饰的对象，可以帮助开发人员读取【请求协议包】信息
  • 称为请求对象

作用：

  • 读取请求行信息：
    • request.getRequestURL();
    • request.getMethod();
  • 读取【请求头】或者【请求体】中所有【请求参数名】
    • request.getParameterNames();
  • 读取请求头或者请求体中的参数内容
    • request.getparameter(“请求参数名”)；
    • request.getparameterValue（“请求参数名”）;
      • 同一参数可能对应 多个值
      • http://localhost:8080/myWeb?uname=mike&uname=rose

请求对象和响应对象的生命周期

  • 都是有Tomcat负责创建的
  • 每当Tomcat收到一个【请求协议包】时，就会为这个请求协议包创建一对【请求对象和响应对象】
  • 一次请求对应一对【请求对象和响应对象】
  • 在Tomcat中调用servlet接口时，负责将本次的请求对象和响应对象作为参数传入服务方法中
  • 在服务方法完毕之后，Tomcat销毁本次请求相关的request和response
  • Tomcat负责将响应包推送到浏览器中

中文请求参数乱码问题

  • 与浏览器采用的请求方式不同有关
    • GET方法，请求参数在请求头存放，请求头内容由Tomcat负责解析，Tomcat默认字符集UTF-8
    • POST方法，请求参数在请求体中存放，请求体中的内容有请求对象request解析，默认字符集不能解析中文字符集
  • 解决方案：在 request对象解析请求体中的数据之前，设置字符集为UTF-8
    • request.setCharacterEncoding(“UTF-8”);

ServletContext接口

【介绍】

  • Servlet规范中的一个接口，接口实现类有Tomcat负责提供
  • 负责为当前工程中所有的Servlet提供共享数据
  • 开发人员习惯将ServletContext接口修饰对象称为全局作用域对象

【生命周期】

  • 在Tomcat启动时，由Tomcat为当前网站来创建
  • 一个网站只能有一个全局作用域对象
  • 从网站启动开始一直到网站关闭，有Tomcat负责销毁当前网站的全局作用域对象

【使用方法】

  • 如何得到当前工程的全局作用域对象？

    • ServletContext servletContext = request.getServletContext();
<!--￼2-->

  • 在网站启动时，Tomcat首先为当前网站创建一个全局作用域对象，Tomcat将web.xml配置共享数据添加到全局作用域对象。

  • 在网站运行时：

    servletContext.getInitParameter("共享数据名称"); 来得到数据

  • 此时共享数据只能读取使用，不能进行修改

  • 【方法二】在Tomcat运行时，可以将某一个Servlet提供的数据保存到全局作用域对象，供其他Servlet使用。

    • AServlet：提供数据

      servletContext.setAttribute(“共享数据名称”，共享数据);

    • BServlet：读取数据

      servletContext.getAttribute(“共享数据”)

    • 此方法可以修改

会话

  • 浏览器和服务端的一次完整的交流
  • 一次会话中可能包含多次请求与响应
  • 在一次会话过程中同一个浏览器往往要访问多个Servlet
  • 在会话过程中，需要在访问的Servlet之间进行数据共享
  • 使用的工具：cookie 和 HttpSession

Cookie

  • Cookie是Servlet规范提供的一个工具类
  • Cookie存储在浏览器上，保存本次会话的私人数据
  • 【工作原理】
    • 首次请求和响应中，cookie通过在响应包的响应头中传回浏览器；
    • 同一会话中下一次请求浏览器将cookie在请求头中再次传回服务端。
  • 【Cookie使用】
    • 创建Cookie
      • Cookie c1 = new Cookie（“key”，“共享数据”）；
      • 【注意】一个cookie只能有一个键值对；
      • 【注意】cookie对象只能存储String类型共享数据
    • 如何将cookie放入响应头？
      • response.addCookie(c1);
    • 如何读取请求头中的cookie？
      • cookie array[] = request.getCookies();

  • 【生命周期】

    • 创建 通过new

    • 销毁

      • 默认情况下，cookie是保存在浏览器的内存中，所以浏览器在关闭时，cookie会被windows销毁
      • 人工干预下，要求cookie保存到客户端计算机的硬盘上，可以制定在硬盘上的存活时间

      ​ 在制定时间内，关闭浏览器或者关闭计算机都不会销毁

      ​ cookie.setMaxAge(以秒为单位)；

  • 【缺点】

    • 一个cookie对象只能存放一个键值对，如果存储的数据较多，需要管理多个cookie
    • cookie只能存放String类型数据，无法存放高级类型数据如list、map

HTTPSession接口

  • 来自Servlet规范中的接口，接口实现类来自于Tomcat
  • 为参与同一个会话的Servlet之间提供共享数据
  • 称为会话作用域对象，也可以叫做session对象

  • 【Httpsession与cookie的区别】

    • 存储位置不同：cookie存储在客户端浏览器内存中，或硬盘上，

      ​ httpsession存储在服务端计算机的内存上

    • 存储内容不同：cookie只能存储String类型数据

      ​ httpsession可以存储任意类型数据

    • 存储数量不同：cookie只可以存储一个键值对

      ​ httpsession可以存储任意个数键值对

  • 【创建方法】

    • 浏览器来访时，Tomcat不会主动为浏览器创建对应的HttpSession对象
    • 只有浏览器访问的Servlet向Tomcat提出要求时，才会为当前浏览器创建HttpSession对象
    • 创建HttpSession对象
      • request.getSession();【有无参数、true、false】三种情况
      • 【无参数时/true】：如果当前浏览器在服务端已经拥有一个session对象，此时Tomcat需要将这个session对象进行返回；若没有对象，则创建一个全新的session对象返回。
      • 【false】: 如果当前浏览器拥有对象，则返回；若没有session对象则返回一个null；
      • 【注意】：如果来访的用户身份已经确认合法，则可以用前两种，若来访用户不合法或未确认，则使用false；

  • 浏览器如何与session关联

    Tomcat创建一个session对象后，为session对象生成一个唯一的编号然后将这个编号保存到cookie中，推送到当前的浏览器内存中；

    等浏览器再次发送请求时，Tomcat可以通过读取浏览器返回的cookie来判断浏览器在服务端中是否有session对象

  • 【使用方法】

    在本次会话过程中，AServlet需要为本次会话中其他Servlet提供共享数据

    ​ session.setAttribute（“key”，共享数据）；

    BServlet想得到session数据

    ​ session.getAttribute（“key”）；

HTTP状态码

  • http状态码是三位数字组成的符号

  • http状态码由Tomcat负责创建生成

  • http状态码由Tomcat写入到状态行

  • http状态码通知浏览器在接收到响应包之后的行为

  • http状态码通知浏览器，服务端无法提供本次服务的原因

  • 【分类】

    • 1XX：通知浏览器本次返回的资源文件并不完整，需要浏览器继续向服务端发送请求

    • 2XX：通知浏览器，服务端将一个完整的文件推送给浏览器。

      ​ 200：浏览器与服务端进行了一次完美通信

    • 3XX：服务端推送给浏览器的是一个网址，要求浏览器在接收到这个网址后，立刻自动向这个网 址发送请求。response.sendRedirect("地址")；

    • 4XX：通知浏览器本次未能提供服务，是由于服务端没有相应的资源文件

      • 404：服务端没有相应的资源文件
      • 405：服务端由本次请求对应的Servlet，但是Servlet不负责处理浏览器当前的请求方式；浏览器使用get请求方式，但是对应的Servlet中没有重写doGet方法
      • 5XX：通知浏览器本次未能提供服务，是由于被调用的Servlet在运行时抛出异常

多个Servlet处理同一请求方案

【前提】：

  • 一个Servlet只负责实现一个功能
  • 浏览器在一次请求中，只能请求一个资源文件
  • 如果浏览器的请求需要由多个Servlet来协同完成，需要用户多次通过手动提交请求来完成任务，这样降低服务质量

【解决方案】

  • 重定向
  • 请求转发

重定向方案

【原理】：

  • 在第一个Servlet工作结束后，将第二个Servlet地址推送给浏览器，由用户浏览器根据这个地址来自动的进行第二次请求

【使用方法】：

  • response.sendRedirect(); 网址必须从网站的根目录写起

【重定向方案特征】

  • 【重定向发生位置】
    • 在客户端浏览器上
  • 【浏览器发送次数】
    • 多次
  • 地址栏发生变化，变为最后一次浏览器访问的资源地址
  • 通过重定向访问的资源文件，通过地址栏命令浏览器发送请求，所以必须采用【GET请求方式】
  • 【重定向时，访问的资源文件范围】：访问的资源文件可以是同一个网站的资源也可以是其他网站的资源

【重定向方案使用场景】

  • 添加/删除/更新功能Servlet 调用 查询Servlet

请求转发方案

【原理】：

​ 在第一个Servlet工作完毕以后，代替当前浏览器向Tomcat申请调用第二个Servlet

​ Tomcat在接收到申请之后，调用第二个Servlet来完成本次请求中的剩余任务

【请求转发使用方法】：

  • 创建资源申请报告对象

    requestDispatcher = request.getRequestDispatcher("下个Servlet地址");

    网址不能写网站的根目录，从内部资源文件开始写

  • 申请报告推送给Tomcat，同时将第一个request和response同时交给Tomcat

    requestDispatcher.forward(request,response);

  • Tomcat在接收到报告之后就会继续调用下一个Servlet

【为什么要把request和response交给Tomcat】：

  • 调用第二个Servlet需要request和response。

【请求转发特征】：

  • 【发生位置】：发生在服务端
  • 【浏览器发送次数】：一次
  • 地址栏不会发生变化，还是第一次Servlet的资源地址
  • 只能访问同一个网站内部资源文件
  • 与浏览器第一次发送请求时的请求方式是一样的，参与请求转发的所有Servlet接受的请求方式是一样的
  • 【请求转发过程中的数据共享】
    • 可以使用全局作用域对象 ServletContext
    • 可以使用会话作用域对象httpsession
    • 可以使用请求作用域对象request

【请求转发适合场景】：

  • 不适合的场景，重定向方案适合的请求转发都不适合
  • 查询Servlet调用JSP时。
